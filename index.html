public List subList(int fromIndex, int toIndex)

源码

继承的文档

返回list的一部分，索引值区间：[fromIndex,toIndex)。特别的，若fromIndex==toIndex，则返回空List（不是null）。

返回的list是原list的一个视图，前者是后者的一部分，并未从后者中分离出去。因此作用在前者之上的修改会反映在后者上，反之亦然。

使用本方法的好处在于，当我们操作list的一部分时，可以直接在这一部分内部操作（仿佛这就是一个新的list），而无需被各种索引的边界分散过多的注意。

若在生成视图后原list发生了结构性的变化（例如长度发生了变化）则已生成的视图将被重置为未定义的状态。

描述

返回的视图在其私有域中存储相对于abstractList的偏移量，存储视图长度（该长度在视图生命周期中可变），存储abstractList的modCount（即abstractList发生结构性变化的次数）。

视图有两个变种：若abstractList实现了java.util.RandomAccess接口，则返回的视图将基于RandomAccessSubList构造；反之，返回的视图将基于SubList构造。

在SubList及RandomAccessSubList中：

在越界检查及偏移量调整后，视图的set(int index, E element)，get(int index)，add(int index, E element)，remove(int index)，addAll(int index, Collection<? extends E> c)，removeRange(int fromIndex, int toIndex)均基于abstractList计算。addAll(Collection<? extends E> c)是通过addAll(size, c)实现的。

listIterator(final int index)返回一个封装的对象，该对象基于abstractList的listIterator，并附加视图的偏移量。iterator()是通过调用listIterator()实现的。size()是通过调用size字段实现的。

所有方法都会检查abstractList当前modCount值（abstractList发生结构性变化的次数）是否与视图的预期相等。若不相等则抛出ConcurrentModificationException。

该方法会抛出如下异常：

  • IndexOutOfBoundsException：索引越界(fromIndex < 0 || toIndex > size)。
  • IllegalArgumentException：fromIndex > toIndex。

源码

描述

比较o与abstractList是否相等。当且仅当o同样是一个list，o与abstractList长度相同，o与abstractList所有对应位置的元素均相等（若设e1为o中的某个元素，e2为abstractList中对应相同位置的元素，若有(e1==null ? e2==null : e1.equals(e2))则有e1与e2相等）。换句话说，两个list相等的条件为：在相同的位置包含相同的元素。

protected void removeRange(int fromIndex, int toIndex)

源码

描述

移除abstractList索引区间为[fromIndex,toIndex)的全部元素。并将后续元素左移补位。本方法减少了abstractList中(toIndex - fromIndex)个元素，若有toIndex==fromIndex，则本方法实际并未对abstractList产生影响。

本方法会被abstractList及其视图的clear()调用。利用list的内部实现重写本方法可大幅提高clear()的性能。

本方法首先得到一个listIterator，并将初始游标定位于[fromIndex-1,fromIndex]。随后反复调用listIterator.next()及listIterator.remove()直到指定范围内的元素全部被移除。

注意：若listIterator.remove()需要线性阶时间，则本方法需要平方阶时间。

protected transient int modCount = 0

源码

描述

abstractList发生结构性变化的次数。结构性变化是指改变abstractList长度，或是其他会使迭代器结果混乱的变化。

本字段将被iterator()返回的iterator及listIterator()返回的listIterator使用。若本字段发生了 iterator/listIterator 所没有预期到的变化，则在调用 iterator/listIterator 的next()，remove()，previous()，set(E e)，add(E e)fail-fast时将抛出ConcurrentModificationException。

在迭代过程中，若检测到并发性变化，则直接判定为失败并抛出异常（fail-fast），而不会去进一步检测所发生的并发性变化是否真的会对迭代造成影响（non-deterministic）。

子类可自行选择是否使用本字段。若子类决定继承本类的fail-fast判定，则只需要在会引发结构性变化的方法中增加本字段的大小。本类中已实现的引发结构性变化的方法有add(int index, E element)及remove(int index)。调用一次add(int index, E element)或及remove(int index)只需将本字段自增1，表示发生了一次结构性变化，否则 iterator/listIterator 会抛出错误的ConcurrentModificationException。

若子类不想遵循fail-fast，忽略本字段即可。

综述

Queue中的E表示queue中元素的类型。

queue是一种有操作优先级的集合。

与Collection接口相比，Queue接口提供了额外的插入，取出，校验操作。在操作失败时每个方法的返回值可能为以下两种返回形式之一：

  • 如果操作失败则抛出一个异常。
  • 返回一个特定值（依操作不同或者为null或false）。此种情况实际是特指，对于insert方法而言，若queue为容量受限的queue则会有特殊的实现方式；在多数实现中，insert操作不会失败。
Throws exception Returns special value
Insert boolean add(E e); boolean offer(E e);
Remove E remove(); E poll();
Examine E element(); E peek();

通常情况下（并非强制要求），queue的优先顺序为FIFO（first-in-first-out），或者也可使用比较器设置特定的优先顺序。另一种数据结构Stack的通用优先顺序为LIFO（last-in-first-out）。

无论使用何种优先级的规则，remove()或poll()返回的都是queue的头元素。在FIFO规则的queue中，所有的新元素都会被添加至queue的尾部。其他实现可能会有不同的添加规则，所有实现必须明确指定其优先级的计算方法。

如果可以插入，offer(E e)方法插入元素，反之返回false。这与其继承的Collection接口的add(E e)有所不同。add(E e)在无法插入时会抛出一个unchecked exception。offer(E e)被用于如下情况：插入失败是正常情况而非异常，例如将元素插入固定容量的队列时很容易无法插入。

remove()及poll()用于移除并返回queue的头元素。移除哪个元素（或者说哪个元素才是queue的头元素）依queue的实现而异。remove()及poll()的差异只体现在queue为空时：remove()抛出一个异常，poll()返回null。

element()及peek()返回但并不移除queue的头元素。

Queue接口中并未定义在并发编程中很常见的阻塞队列方法。这些阻塞队列方法会等待元素的出现或者空间的空出，它们被定义在了Queue的直接子类接口BlockingQueue中。

Queue接口的实现通常禁止插入空元素（当然也有例外，例如LinkedList就允许插入空元素）。即便是在例外中，也不应该向这种Queue的实现中插入空元素。因为poll()在queue为空时会返回null，而如果queue中包含空元素，则我们无法判断poll()返回的null所表达的含义到底是queue为空还是queue返回了一个空元素。

在equals(Object o)及hashCode()的设计上，Queue接口的实现在定义相等或计算hash code值时通常不是基于元素本身的类型的，而是基于类Object的。因为元素本身的类并不总能很好的定义同一个元素在不同的队列优先级下的相等性。
